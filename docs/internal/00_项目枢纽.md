# Sit 项目枢纽

本文件是 `sit` 仓库的第一入口。后续处理本目录任务时，先读本文件；若改变项目目标、架构、命令范围、数据结构、验证方式或路线图，先在"处理记录"写明意图，再执行，最后回填结果。

## 1. 项目目标

`sit` 是 **agent-native 的 git**：当多个 agent 并行演化同一个共享 skill 库时，`sit` 让它们像人类开发者用 git 协作一样可观测、可验证、可回滚、可治理。详见 `docs/research/proposal.md`（2026-05-20）。

- 项目身份只有一个：`sit` CLI。它不重写 Git，而是在 Git 之上增加 Skill 语义层 + multi-agent skill 协作协议（FORK / EVOLVE / PROPOSE / VERIFY / MERGE / REVERT / RETIRE / GOVERN）。
- 线上协作直接复用 GitHub / GitLab。`sit` 只负责让那一层"看得懂 Skill"——通过稳定 JSON 契约（`sit.info.v1` / `sit.pr-summary.v1` / `sit.report.v1` / `sit.test.v1`）暴露语义 diff、PR 摘要、测试报告，供任意 Web/CI/IDE 消费。

项目立足两类受众，并以第二类作为研究差异化：

- **人类团队**：围绕 Skill Package 做版本控制、评审、测试、发布、复现和协作（已通过 `paper-webpage-builder` 等真实试点验证一阶闭环）。
- **多 agent 系统**：N 个 agent 共享演化一个 skill 库，`sit` 提供版本历史、语义 diff、行为回归 gate、回滚、依赖追踪、PR/CI 反馈，使任意 intra-agent skill 算法接入即获得协作基础设施。这是 proposal 的核心论点：**与具体 agent 算法正交的协作基础设施**。

项目的核心不是"保存历史"——Git 已经能做到。核心是回答：**能否为多 agent skill 协作构造一套 agent-native 的 git，使之像人类协作一样高效、可控？**

## 2. 控制论式项目模型

本项目采用《工程控制论》迁移出的思考方式设计：把 Skill 协作看作一个受控系统。模型对人类团队和多 agent 系统同构适用——区别只在反馈动作的执行主体（人 vs agent）。

| 控制论概念 | `sit` 中的含义 |
|---|---|
| 受控对象 | Skill Package 的生命周期：编辑、测试、diff、review、release、复现；多 agent 场景下扩展为共享 skill 库的并发演化轨迹 |
| 状态 | 当前 skill.yaml、prompts、schemas、tests、deps、reports、Git ref；agent 视角下还包含每个 agent 本地缓存与 reputation |
| 输入 | 人的编辑、分支、PR、依赖升级、运行环境变化、模型版本变化；agent 的 fork / evolve / propose 决策 |
| 输出 | 新版本 Skill、结构化 diff、测试结果、review 摘要、release tag；agent 间传递的 `sit.pr-summary.v1` 等 JSON 契约 |
| 误差信号 | schema breaking change、golden regression、prompt 行为漂移、依赖不兼容、评审意见；多 agent 场景下还包含 bad-skill 注入、异构演化冲突、有害 propose 占比 |
| 反馈动作 | validate、test、diff、pr-summary、release bump、回滚、补充 golden case；多 agent 场景下还包含 `git revert` 精确回滚、reverse-deps 限制 blast radius、reviewer agent 投票 |
| 稳定性约束 | 不破坏已发布版本、不让 schema 变更静默传播、不让 prompt 行为漂移不可见；多 agent 场景下还包含 RETIRE 可观测、治理不显著降低吞吐 |
| 最优控制 | 在 Git 兼容、实现成本、协作价值之间选择当前最能降低风险的下一步；多 agent 场景下还要在样本效率、协调成本、库质量三维 Pareto 前沿上权衡 |

第一阶段的 `sit` 不追求平台完整性，而追求建立一个**可观测、可反馈、可验证、可复现的本地控制回路**——这是人类团队 PR 闭环和多 agent driver 闭环共同的底座。

## 3. 当前路线

`sit` 是 agent-native 的 git；线上协作直接复用 GitHub/GitLab，**不另起一个对标 GitHub 的产品**。叠加 proposal 给出的实证验证层：

```text
sit CLI -> 封装 Git + Skill 语义层 + multi-agent 协议动词
GitHub/GitLab -> 线上协作承载（人）+ 远端中央仓库（agent）
multi-agent driver -> 实验代码层，在 sit 之上跑 H1/H2/H3/H4 实验
JSON 契约 (sit.*.v1) -> 让任意 Web/CI/IDE 集成消费 sit 输出
```

当前不做：

- 自研对象存储。
- 自研分支、提交、合并和远程协议。
- Web Editor / 自建 Web 协作平台 / Marketplace / Registry（任何对标 GitHub 的产品都不做）。
- 自动语义 merge（多 agent 场景下冲突由 git 报、由 agent 解决）。
- 把 `sit` 变成 multi-agent 框架本身（driver、reviewer agent、reputation ledger 留在 `experiments/` 外部组件）。
- 提出新 LLM / 新 agent 算法。

当前必须做：

- 标准 Skill Package 文件结构。
- `sit init` 创建包和 GitHub CI/PR 模板。
- `sit validate` 让结构可观测。
- `sit diff` 让变更语义可观测（含 schema、prompt、scripts、resources、status 生命周期）。
- `sit test` 让行为回归成为反馈信号。
- `sit pr-summary` 让 GitHub PR 获得 Skill 语义摘要，同时是 agent 间传递的"提案对象"。
- `sit release` 让版本变化和 Git tag 对齐，并触发反向依赖告警。
- 稳定的 JSON 契约（`sit.info.v1` / `sit.pr-summary.v1` / `sit.report.v1` / `sit.test.v1`），让 agent driver 可机读消费。
- 协议动词覆盖：FORK / EVOLVE / PROPOSE / VERIFY / MERGE / REVERT / RETIRE / GOVERN 在 sit + git 上有对应实现（详见 `docs/research/sit-multi-agent-mapping.md`）。
- multi-agent driver 第一版与 H2 lead 实验，作为 proposal 主张的实证基线。

## 6. 当前状态

已有可用 CLI MVP：

- `sit init`
- `sit onboard`
- `sit status`
- `sit info`
- `sit doctor`
- `sit validate`
- `sit test`
- `sit diff`
- `sit report`
- `sit pr-summary`
- `sit release`
- `sit deps check`
- `sit add/commit/push/pull/branch/checkout/log`

这些能力是第一版控制回路的雏形，但还不足以完成 `skill git.md` 的目标。缺口主要是：

- Git ref 语义已支持第一版：`sit diff main..HEAD`、`sit pr-summary main..HEAD`、`sit report --compare main..HEAD`。
- `sit diff` 和 `sit pr-summary` 已支持 `--format text|markdown|json` 第一版。
- `sit info` 已支持 `--format text|json` 第一版，可展示 package、Git、文件、validate/test 和最近 report 状态。
- `sit onboard` 已支持已有 `SKILL.md` 项目接入第一版，可补齐 `skill.yaml`、schemas、golden case、GitHub Actions、PR 模板和 onboarding reports，默认不覆盖已有文件，并以 `sit doctor` 状态收尾。
- `sit standardize` 已支持把普通 prompt / `SKILL.md` 项目整理成标准 Skill Package：补齐 `skill.yaml`、`prompts/`、input/output schema、golden tests、GitHub Actions 和标准化报告；生成 schema/golden 为可编辑草稿。
- `sit doctor` 已支持 `--format text|json` 第一版，可检查已有 Skill 的 Git/GitHub/manifest/validate/test/workflow/report 接入状态。
- `sit diff` 已支持递归 schema diff 第一版：nested object、required、type、enum、items、additionalProperties 和常见边界约束。
- `sit diff` 已支持复杂 schema diff 第一版：`oneOf`、`allOf` 和本地 `$ref`，可解析 `#/definitions/...` 与 `#/$defs/...`。
- `sit test` 已支持 golden case 的 `schema_only`、`exact`、`partial`、`contains` match mode；`sit test --run` 已支持通过 `commands.run_case` 或 `--runner` 运行 Skill case 生成 actual 后做行为回归比较。
- `sit commit` / `sit release` 已支持 version gate：在有 Git HEAD 基线时阻断低于语义 diff 所需级别的版本变化，可用 `--no-version-gate` 显式绕过。
- `sit test` / `sit report` / `sit info` 已具备机器可读 JSON 输出第一版：`sit.test.v1`、`sit.report.v1`、`sit.info.v1`。
- `sit report --format html` 已支持静态可视化报告，包含风险筛选、长 diff 折叠/展开、复杂 schema path 展示；example 已生成 `examples/paper-taxonomy-mapper-v0.2.0/reports/visual-v0.1.0-to-v0.2.0.html`。
- `sit ci-summary` 已支持从 report payload 生成 GitHub Actions Markdown summary，支持自定义 baseline/head ref、package 子目录和 JSON/Markdown/HTML artifact 输出；`sit init` 生成的 CI workflow 会写入 `$GITHUB_STEP_SUMMARY` 并上传失败复盘 artifact。
- version gate 阻断信息已包含原因、修复建议、绕过提示和关键语义变更；release note/report 已包含风险、version gate、验证/测试和语义变更摘要。
- `sit diff` 已支持 scripts/assets/references 资源变更事件；脚本变更会标记 `review-required/minor`，并提示 runner 或专项测试覆盖。
- `sit diff --prompt` 已支持 prompt/reference unified text diff；默认 text/markdown 输出会展示增删行数、关键标题和模板变量摘要；`--format plain` 保留旧逐行输出。
- `sit report --compare <git-range>` 的 Markdown/HTML 展示已隐藏 `/tmp/sit-ref-*` 快照路径，改用 `HEAD~1` / `HEAD` 等稳定 source 标签；JSON payload 仍保留真实 root。
- `sit release --bundle` 已支持生成 `dist/<skill>-v<version>.tar.gz`，包内包含 `manifest.json`、`reproduce.sh`、核心包文件、reports 和 sha256 清单。
- `sit release` 的 CHANGELOG/release report 已按 Breaking / Changes / Fixes / Risk / Reproduce 结构从 semantic diff 自动生成，并会扫描同级目录反向依赖，提示 compatible / review / incompatible。
- `sit deps check` 已支持 `deps.yaml` 本地 path 依赖、版本范围检查和 baseline schema 兼容性预警；`sit commit` 会打印依赖 warning 但不阻断。
- `scripts/build_binary.py --dry-run` 已提供 PyInstaller 独立二进制构建入口；当前环境未安装 PyInstaller，尚未产出真实 binary。
- `demo.html` 已对齐 resource-aware diff 和 prompt/reference text summary，可展示 SCRIPT/ASSET/REFERENCE 事件预览。
- `paper-webpage-builder` 真实试点已接入第一版 runner-backed behavior regression：`commands.run_case` 调用 `scripts/sit_run_case.py`，3 条 golden case 升级为 `partial` match 并保留静态 `actual` fixture，GitHub Actions 会运行 `sit test --run` 并上传 `sit-test-run.txt` artifact。

### 7.2 当前最高优先级：让语义 diff 覆盖真实变更

#### F3: 脚本和资源文件纳入语义 diff

**误差信号**：`paper-webpage-builder@0.4.0` 闭环中，`scripts/scan_paper.py` 发生了会影响行为的修复，但 `sit diff` 只显示 prompt/reference 变化，没有显示脚本变化。

**用户价值**：reviewer 能看到“工具行为也变了”，不会只审 prompt/schema。

**要做**：
- [x] 明确 `skill.yaml` 中 scripts/assets/references 的可观测资源范围，或默认扫描常见目录：`scripts/`、`assets/`、`references/`。
- [x] `sit diff` 输出 `SCRIPT changed` / `ASSET changed` / `REFERENCE changed` 事件。
- [x] 脚本变更默认标记 `review-required`，并提示需要 runner 或专项测试覆盖。
- [x] `sit report`、`ci-summary`、`pr-summary` 同步展示这些事件。
- [x] 增加回归测试：修改 `scripts/*.py` 时，`sit diff HEAD~1..HEAD --format json` 必须包含 resource/script event。

**完成标准**：
- 在 `paper-webpage-builder@a4d0f92` 这类变更中，`sit diff` 能同时显示 prompt 变化和 `scripts/scan_paper.py` 变化。
- `sit commit` 的 version gate 继续给出 `review-required/minor`。

#### F4: Git range report 中的临时路径清理

**误差信号**：`sit report --compare HEAD~1..HEAD` 会在报告正文中展示 `/tmp/sit-ref-...` 快照路径，复现命令是对的，但报告可读性差。

**用户价值**：报告给 reviewer 看时，路径应该表达为 `HEAD~1..HEAD`、`baseline`、`current`，而不是临时目录实现细节。

**要做**：
- [x] report payload 中保留真实 package root，但展示层使用 source/ref label。
- [x] Markdown/HTML report 的 Package、Validation 和 Reproduce 区域隐藏 `/tmp/sit-ref-*`。
- [x] 增加测试覆盖 Git range report 的展示路径。

## 8. 产品分发与用户体验规划

当前路线只到 CLI + 二进制分发 + VS Code 插件三层；线上协作仍由 GitHub/GitLab 承载，`sit` 通过 JSON 契约喂数据。下文 8.3 Step 3 / 8.6 / 8.7 中的 Web 平台设想保留为历史讨论，**不再作为执行项**；`demo.html` 与 `prompt-lifecycle-demo.html` 仍作为静态体验验证页，不延伸为产品。

### 8.1 核心问题

当前 sit 的用户获取路径太长：理解 Skill Package 概念 → 按规定结构组织文件 → 学一套新命令 → 改变现有工作流。这相当于"你先按我的规矩来，然后我才帮你"。正确的思路是先给价值，再要求结构。

### 8.2 Git 的成功模式

Git 本身是 C 语言写的系统级二进制程序，不依赖任何运行时，装了就能用。但 Git 的"方便"不来自 CLI 本身，而来自生态：

- GitHub/GitLab Web UI — 看 diff、review PR
- VS Code 内置 Git 面板 — 点按钮就能 commit/push
- GitHub Desktop — 给不碰命令行的人用
- CI 集成 — push 了自动跑，结果自动显示

VS Code 的"源代码管理"面板不是重新实现 git，它只是调用系统里的 `git` 命令，把结果渲染成图形界面。

### 8.3 sit 要达到同样体验的路径

#### Step 1：独立二进制分发（消除 Python 依赖）

当前 sit 是 Python 包，用户必须有 Python 3.10+ 并 `pip install`。要像 git 一样"装了就能用"，需要打包成独立二进制。

| 方案 | 用户体验 | 实现成本 |
|------|----------|----------|
| PyPI（当前） | `pip install sit-toolkit`，需要 Python | 已完成 |
| PyInstaller 打包 | 下载一个文件放到 PATH，直接能用 | 1-2 天 |
| Homebrew formula | `brew install sit` | 1 天 |
| Go/Rust 重写 | 零依赖单文件二进制，像 git 一样 | 几周到几月 |

短期方案：PyInstaller 打包。长期方案：考虑 Rust 重写核心引擎。

#### Step 2：VS Code 插件（覆盖开发者日常）

VS Code 插件用 TypeScript 写，底层调用 `sit` CLI 命令（和 VS Code 调用 git 完全一样的模式）：

```
VS Code Extension
    │
    ├── 检测工作区是否有 skill.yaml → 激活 Skill 面板
    │
    ├── 调用 sit info --format json → 显示 Skill 状态
    ├── 调用 sit validate → 实时显示校验结果（红色波浪线）
    ├── 调用 sit diff --format json → 渲染语义 diff
    ├── 调用 sit test --format json → 显示测试结果（绿灯/红灯）
    │
    └── 文件保存时自动触发上述命令
```

用户体验：

```
用户在 VS Code 里打开一个 Skill 项目
侧边栏自动出现 "Skill" 面板（和 Git 面板并列）：
  - 当前版本、schema 状态、测试结果
  - 修改 prompt 后实时显示语义 diff
  - 点击"提交"自动跑 validate + test
  - 点击"同步"推送到 GitHub
  - 不需要打开终端，不需要记命令
```

#### Step 3：渐进式 Web 体验（覆盖非技术用户）

针对教师、PM 等不碰终端的用户，提供 Web 界面。但不需要从零建平台，分三步：

**3a. 静态 Web Viewer（1-2 周）**

纯前端页面，用户上传 skill.yaml + prompt 文件 → 看到格式化信息和 diff。不需要后端。让人不装 sit 就能体验核心功能。

**3b. 在线编辑器 + 版本历史（1-2 月）**

用户注册账号，在线编辑 prompt，每次保存自动生成版本，可视化 diff，一键回退，简单测试面板。替代"prompt 存在飞书文档里"的现状。

**3c. 协作 + CI（2-3 月）**

邀请团队成员，变更审批流程，自动测试，和 GitHub 同步。

### 8.4 降低学习成本的具体做法

| 当前门槛 | 解决方案 |
|----------|----------|
| 用户不知道 Skill Package 是什么 | 不要求他知道。上传/打开一个 prompt 文件就能开始 |
| 用户不会写 JSON Schema | 提供模板，或从 prompt 自动推断 schema |
| 用户不会写 golden test | 提供"录制"功能：跑一次，把输出存为 expected |
| 用户不会用 CLI | VS Code 插件覆盖所有核心操作 |
| 用户不理解 breaking change | 用人话说："这个改动会让依赖你的人出错" |

### 8.5 渐进式功能解锁（用户不需要一次学完）

| 用户阶段 | 他做的事 | sit 提供的价值 |
|----------|----------|---------------|
| 第 1 分钟 | 打开/粘贴一段 prompt | 自动保存版本，可以回退 |
| 第 5 分钟 | 修改 prompt | 高亮 diff，告诉他改了什么 |
| 第 10 分钟 | 加一个测试 case | 每次改 prompt 自动跑测试，告诉他"这次改动让 case 2 挂了" |
| 第 1 天 | 定义 input/output schema | 自动检测 breaking change |
| 第 1 周 | 邀请同事协作 | PR 流程、review、权限 |
| 第 1 月 | 管理多个 Skill | 依赖图、版本发布、registry |

### 8.8 关键认知

sit 不需要替代 git，它是 git 的扩展。就像 ESLint 是编辑器的扩展（红色波浪线）、Prettier 是保存时的自动格式化。sit 应该是：用户照常用 git，但因为装了 sit（或 VS Code 插件），自动获得 Skill 语义层的保护。理想状态下用户甚至不需要主动"使用 sit"——他编辑文件、保存、提交，sit 在背后自动跑校验、自动标记风险、自动生成摘要。
